So we ran through the lighting network graph, the nodes of vertices, the channels in this graph of edges and at the moment nodes need to keep this graph in memory so they can route quickly if you stored in your database it’s gonna slow you down very much and that’s starting in about 40 Meg’s which is acceptable at the current state of the network but we do expect this to grow. Node announcement

The next message that we use is something called a node announcement. This is a bit more sort of miscellaneous. It allows you to add further information to the channel announcements which contain your node pub key and we only take these into account according to the spec if these nodes already have a channel open.

There’s no point for you keeping track of a node if it doesn’t have a channel open it doesn’t serve you at all it has no purpose. and it’s also a potential cost vector so we just discard these messages and do not forward them in the network. These messages also indicate it’s willingness to accept inbound connections, so ipv4 six orto addresses which aren’t really related to routing but it does provide you with some additional information. Channel updates

And finally we look at channel updates so with no announcements and channel announcements we can build what the graph looks like. This also contains the time-lock information that you that you wish to accept over this channel and the limits that you play some HTLCs the minimum amounts the maximum amounts and a total number of HTLCs is you’re willing to route through this channel and these again are set per node in a channel so a channel has direction you can move funds back and forth between the two peers and each node has the right to determine their own fees in their own time lock according to their risk preference.

These messages can also be used to keep your node alive, so if you haven’t done anything for two weeks your tunnel will be pruned from the graph in most implementations so you can send out one of these to keep alive. It’s a bit of an issue at the moment. One of the first problems and one of the biggest problems we face in router in the night network is that of unknown child balances so we advertise in that node in our channel announcing the capacity but not the balance. So this is when you calculate a route that includes that channel edge but it doesn’t actually have any Bitcoin on the side that you’re wishing to move across. And this means that your routing attempt will fail. So what a lot of the implementations do is they implement iterative routing. This means that you retry again, so you try to calculate a new route and either exclude that edge entirely from your graph temporarily, or you don’t ban them for having unbalanced channels. There are a few solutions here as well. The first is to worry for updates so when you receive a policy failure you’ll know that you’ve got the fee or the time lock wrong or the number of HTLCs so you can query for that specific nodes update and then get it. However the issue still has the issue where you do have to recalculate the route you still have to get that information and then retry and delay the user experience further. Another big one would be to have some gossip improvements so the spec has agreed to move to inventory based gossip which will be much more efficient in terms of bandwidth. Another future problem for the Lightning Network is that of client Rotten. We could see that the memory requirement for the graph grows too big if we end up with a very large network graph. We could face bandwidth constraints in very small nodes that don’t want to sync up the whole graph or even parts of the graph and we could struggle with route computations. Some of the solutions and trade-offs here are the first one is to prune the graph and I think this is pretty low hanging fruit and there’s a lot of space for optimization especially with the current state of the network which is a 40 megabyte graph it is still pretty manageable as is. So you can do so by removing small channels already bad channels with nodes that aren’t online. Another solution that people are starting to look at is the idea of offloading computation. So rather than like clients having to do all of this computation by themselves they offload that to more resource intensive clients who do that computation on their behalf in exchange for fees.
